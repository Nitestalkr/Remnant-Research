# GRAO Persistent Failure Analysis

## Overview

From Round 13 through Round 39, exactly **10 failure gradients** have persisted across every cycle without resolution. This document investigates their root cause, classifies the failure modes, and proposes resolution paths.

**Key observation:** The absolute number of failures has remained constant at 10 while the total trace count grew from 19 to 137. This means the failures are not a growing problem but a **fixed set of traces that consistently fail gradient derivation**.

## Root Cause Analysis

### Primary Cause: Trace Metadata Compliance

In Round 13, the bottleneck was explicitly identified:

> "Trace metadata fields (type/source/target) unknown in 10/19 traces"

The 10 persistent failures are traces that lack required metadata fields:
- `signal_type` — not classified as agent/research/stability/experience
- `source` — missing trace source identifier
- `target` — missing trace destination

Without these fields, `gradient-deriver.js` cannot classify the trace into a cluster, so it fails to compute a gradient.

### Why They Persist

1. **Trace origin** — Some traces are generated by legacy cron jobs or system events that predate the current trace schema
2. **Schema drift** — The trace schema evolved over time; older traces don't match current requirements
3. **No cleanup mechanism** — There's no automated process to identify and fix non-compliant traces
4. **Low priority** — As the total trace count grew, the failure rate dropped (10/137 = 7.3%), making the issue less urgent

### Classification of the 10 Failures

| Category | Estimated Count | Description |
|----------|----------------|-------------|
| **Legacy traces** | ~4 | Traces from pre-schema cron jobs |
| **Incomplete traces** | ~3 | Traces missing one or more metadata fields |
| **System traces** | ~2 | Internal system events not designed for GRAO |
| **Corrupted traces** | ~1 | Traces with malformed JSON or invalid data |

## Resolution Paths

### Path 1: Trace Schema Enforcement (Recommended)

Add schema validation to `trace-collector.js` that rejects non-compliant traces early:

```javascript
function validateTraceSchema(trace) {
  const required = ['signal_type', 'source', 'timestamp'];
  for (const field of required) {
    if (!trace[field]) {
      return { valid: false, reason: `Missing field: ${field}` };
    }
  }
  return { valid: true };
}
```

**Pros:** Prevents new non-compliant traces from entering the system
**Cons:** Doesn't fix the existing 10 failures

### Path 2: Trace Remediation

Add a remediation step to `trace-collector.js` that attempts to infer missing fields:

```javascript
function remediateTrace(trace) {
  if (!trace.signal_type) {
    trace.signal_type = inferSignalType(trace); // Heuristic classification
  }
  if (!trace.source) {
    trace.source = inferSource(trace); // Heuristic source detection
  }
  return trace;
}
```

**Pros:** Fixes the existing 10 failures
**Cons:** Heuristic inference may be incorrect; adds complexity

### Path 3: Graceful Degradation

Modify `gradient-deriver.js` to handle incomplete traces more gracefully:

```javascript
if (!trace.signal_type || !trace.source) {
  // Log warning, attempt best-effort classification
  const gradient = computeBestEffortGradient(trace);
  if (gradient) {
    storeGradient(gradient, true); // true = best-effort
  }
}
```

**Pros:** No data loss; all traces contribute something
**Cons:** Lower quality gradients; harder to distinguish from real failures

### Path 4: Archive and Replace

Archive the 10 non-compliant traces and replace them with remediated versions:

1. Identify the 10 traces by content hash
2. Archive to `gradients/archive/` with failure reason
3. Create remediated versions with inferred metadata
4. Store in `gradients/` as corrected traces

**Pros:** Clean resolution; documented history
**Cons:** Manual process; needs to be repeated as new non-compliant traces appear

## Research Questions

The persistent failures are not just a bug — they're a research question:

1. **What fraction of system traces are GRAO-incompatible?**
   The 10/137 ratio (7.3%) suggests most traces are compatible, but the absolute number is constant. Is there a hard limit on compatible trace sources?

2. **Does the fixed failure count indicate a structural limitation?**
   If the failures are always from the same sources (legacy cron jobs, system events), this may indicate a need for GRAO-aware system instrumentation.

3. **Should GRAO have a "best-effort" mode?**
   The current strict schema enforcement means 10 traces contribute nothing. A graceful degradation mode would capture more signal at the cost of lower quality.

4. **What is the optimal trace schema?**
   The current schema (signal_type, source, target) may not cover all trace types. Is a more flexible schema needed?

---

## Recommendation

**Short-term:** Implement Path 1 (schema enforcement) to prevent new non-compliant traces.

**Medium-term:** Implement Path 2 (remediation) to fix the existing 10 failures.

**Long-term:** Investigate the research questions above to determine if GRAO needs a more flexible trace schema or a best-effort mode.

The 10 persistent failures are a known gap, not a critical issue. They represent 7.3% of the current trace pool and are not growing. The priority is preventing new failures, not fixing the existing ones immediately.
